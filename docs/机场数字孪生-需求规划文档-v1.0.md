## 2. 用户与任务分析（D1）

### 2.1 核心任务与成功条件

**T1 现场演示**（最高频）："我打开这个页面，5 分钟内向观众讲清一个机场运控概念，中途不出岔子。"

| 成功条件 | 反推需求 | 现状 |
|---|---|---|
| 任何网络环境下页面数秒内出画面 | 依赖可自托管或有完整性回退；失败时给诊断页而非白屏 | ❌ Three.js 仅 CDN，断网/内网即白屏（规格 §9） |
| 讲解状态可提前布置、可复现 | 配置与面板布局持久化；场景可编码进 URL 分享 | ❌ 除语言外零持久化（KI-9） |
| 演示中途误刷新可在 30 秒内恢复 | 同上（持久化 + URL） | ❌ 全部重置 |
| 出了错演示者第一时间知道 | 全局错误捕获 + 健康横幅 | ❌ 零错误上报（KI-10） |
| 投影/核显机器不掉帧 | 性能档位（阴影/像素比降级） | ⚠️ 无档位，阴影 4096² 固定 |

**T2 概念教学**："我制造一个情景（关跑道/LVP），让学生看见因果。"

| 成功条件 | 反推需求 | 现状 |
|---|---|---|
| 情景可一键布置 | 预置剧本库（高峰压力/LVP 演练/单跑道日） | ❌ 每次手工点配置 |
| 因果可回看 | 录制回放 | ✅ FR-11（RECALL） |
| 课后可留档/换机重放 | 录制导出后可**导入**重放 | ⚠️ 只有导出，无导入通道 |

**T3 二次消费**（开发者）："我从孪生取标准数据流做自己的分析。"

| 成功条件 | 反推需求 | 现状 |
|---|---|---|
| 有稳定文档化的数据出口 | snapshot 契约 + schemaVersion | ✅ FR 契约 §6，window.__snapshot |
| 契约变更可察觉 | 契约冒烟测试守护字段 | ❌ 无任何测试 |

### 2.2 优先级纠偏结论

现有路线图惯性是"每轮加一个行业件面板"；JTBD 一测即知：**T1 的五个成功条件有四个不成立**，而它是最高频任务。演示可靠性（错误可见、状态可复现、依赖可控）是任务前提，全部判 P0——排在任何新行业件之前。

## 4. 场景与环境分析（D4）

设计基准定为：**中档核显笔记本 + 1080p 投影 + 会场 Wi-Fi（不可信）**。

| 环境压力 | 现状预算 | 派生需求 |
|---|---|---|
| 核显 GPU | 阴影 4096² 固定 + ACES + 全屏 3D，无降级 | PR：性能档位（自动检测掉帧→降阴影/像素比） |
| 会场网络 | Three.js 0.169 CDN 单点，无 SRI 无回退 | PR-P0-3：自托管或双源+完整性校验 |
| 投影色彩 | 深色主题对比度高 | 无需求 |
| 长时间挂机演示 | 环形缓冲有界 ✅；但后台切走时间丢失（KI-2） | PR：可见性恢复提示（"后台 N 分钟，仿真已暂停推进"） |

## 5. 失效模式与可观测性（D3）

核心一问：**哪些失效用户当下察觉不到？** FMEA-lite（现状证据见 Gap 附录 B）：

| 失效 | 用户当下可察觉? | 后果 | 派生需求 |
|---|---|---|---|
| 任意 JS 异常（含 CDN 加载失败） | 否——零捕获零提示 | 白屏或面板悄然停更，演示翻车 | PR-P0-1 全局捕获+健康横幅+诊断页 |
| 标签页后台被节流 | 否——回来只见"好像没动" | 讲解时间线错乱 | PR-P0-1 附属：可见性变化提示 |
| 机位全满丢航班 | 弱——仅日志一行 | FIDS 上航班"消失"，观众困惑 | PR-P1：no_gate 转入等待/改航语义并可视化 |
| 吞吐 KPI 混时基 | 否 | 快进演示时指标失真误导 | PR-P1：全 KPI 统一 sim 时基 |
| 环形缓冲丢早期录制 | 否——回放范围悄然缩短 | "刚才那段"回放不到 | PR-P1：回放条标注已丢弃区间/容量水位 |
| 设置漂移（刷新回默认） | 是但为时已晚 | 复现成本高 | PR-P0-2 |

派生"健康自检"类需求成型为 PR-P0-1（见 §9）。

## 9. P0 缺口专项

### PR-P0-1 错误与健康可观测性

- 任务影响：T1 第 4 条成功条件的唯一载体；当前任何异常对演示者完全静默（KI-10）
- 方案概要:window.onerror + unhandledrejection 挂钩 → 顶部红色健康横幅（计数+最近错误+复制详情）；import map 加载失败时 body 兜底诊断块（静态 HTML 即可渲染）；document.visibilitychange 恢复时横幅提示"后台 N 分钟仿真未推进"
- 破坏性验收：控制台注入 throw；断网刷新页面；后台切走 5 分钟再回——三种情况各自必须在 3 秒内出现对应可视提示，绝不白屏或无声
- 量级：小（单文件 + 少量样式）

### PR-P0-2 演示状态可复现

- 任务影响：T1 第 2/3 条；误刷新 = 演示重来
- 方案概要：配置对象集中化（现散在 _cfg 与各 API 内）→ localStorage 自动存取；encode 成 URL 查询串（?wx=3&rwy=1&gates=8…），load 时优先 URL；面板折叠态一并存
- 破坏性验收：布置 LVP+关 RWY1+8 机位 → 强制刷新 → 页面回到完全一致状态且横幅提示"已恢复上次场景"；把 URL 发到另一浏览器打开 → 同一场景
- 量级：中（触及 main.js 动作分发与各 setter 的幂等性）；含 D6 要求的参数白名单校验

### PR-P0-3 依赖可控

- 任务影响：T1 第 1 条；内网/断网即白屏
- 方案概要：Three.js 0.169 构建物入仓自托管（约 1.2 MB，import map 改本地路径）；或保 CDN 主源 + 本地回退双 importmap + SRI integrity
- 破坏性验收：断网（DevTools offline）刷新——场景照常渲染；篡改本地 three 文件一字节（模拟投毒）且启用 SRI 时——拒载并出诊断页
- 量级：小-中（含部署脚本同步该目录）

## 11. 验收与质量策略（D7）

- 夹具资产：__step 全管线快进（已有）是一切自动验收的地基——PR-P1-1 把它从"控制台手艺"固化为仓库内脚本
- 破坏性验收写法示范（除冰 PR-P2-1 预写）："雪情景下航班过站完成但 HOT 已过期，必须被强制送回除冰坪重除，绝不能带着过期 HOT 起飞"
- 现有人工手段（对抗审查+浏览器实测）定位为兜底，不替代自动回归

## 12. 非目标与已知取舍

| 显式不做 | 理由 |
|---|---|
| 外部真实数据接入/任何网络 IO | 演示定位 + 纯静态约束 C-1；引入即引入后端与合规负担 |
| 多机场 | 单布局已满足教学；布局文件化（PR-P2-4）后自然留有接口 |
| 旅客流/行李流仿真 | 范围锁定"地面运行"；另立项更合适 |
| 封锁 window.__* 调试面 | 演示产品的开放性是特性 |
| 旧浏览器兼容（无 import map 环境） | 常青浏览器已覆盖目标观众；成本不成比例 |
