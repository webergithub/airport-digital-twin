### 1.3 术语

| 术语 | 含义 |
|---|---|

| sim-秒 | 仿真时钟秒。全部运行逻辑用 sim 时钟；时标压缩（过站 60 sim-秒 ≈ 现实 45 分钟量级） |

| A-CDM | 机场协同决策。里程碑：ATA 到场、ALDT 接地、AIBT 靠桥、TOBT 目标撤轮挡、ARDT 就绪、TSAT 开车许可、AOBT 撤轮挡、ATOT 起飞 |

| RECAT-EU | 欧洲尾流重分类间隔标准（本产品压缩为 sim-秒简化矩阵） |
| A-SMGCS | 先进场面活动引导与控制系统。本产品覆盖其监视（雷达）、安全网（RIMCAS）、引导（Follow-the-Greens）三支柱 |

| OOOI | 航空公司电报四事件：Gate OUT／wheels OFF／wheels ON／gate IN |
| ASPM | FAA 航空系统绩效指标库；本产品按其口径出分跑道滑行时间中位数/P90 |
| DCB | 需求-容量平衡：把预测流量投入时间桶对比容量、标记热点 |
| RECALL | 场面运行录制回放（仿 Searidge RECALL） |
| Follow-the-Greens | 滑行道中线绿灯引导 + 红色停止排灯（ADB Safegate 产品形态） |

## 2. 产品概述

### 2.1 产品定位

面向演示与教学的机场地面运行数字孪生：在浏览器里完整跑一个"到港 → 靠桥保障 → 推出 → 起飞"的闭环，并配齐真实机场运控系统的行业件（A-CDM、AMAN/DMAN、A-SMGCS、DCB）。核心场景：

- 行业演示：向非民航观众直观展示机场运控概念（每个面板对应一个真实系统品类）
- 概念教学：A-CDM 里程碑链、尾流间隔、门位扣留节油等机制的可视化讲解
- 情景推演：关跑道／恶劣天气下容量塌缩与热点预测的因果演示

### 2.2 目标用户与典型场景

| 用户 | 典型场景 |
|---|---|
| OPC Studio 站点访客 | 打开即看，自动生成航班流；点机位看保障细节；切英文 |
| 民航科普/教学者 | 展开 AMAN 梯／安全网／DCB 面板逐概念讲解；用情景推演做"关跑道会怎样"实验 |
| 开发者 | 经 window.__snapshot／__api 读标准 JSON 快照做二次消费；F12 调 __step 快进复现 |

### 3.2 拓扑与数据流

控制层每逻辑 tick 由 main.js 驱动：api.update(dt) → scheduler.update(dt) → getSnapshot() 挂到 window.__snapshot → analytics/safetyNet/dcb/runLog 各自消费 → 15 个面板刷新（main.js:216-264）。反向仅两条：算法层自动调参调 scheduler.setInterval（analytics.js:203-208）；UI 动作经回调进 main.js 再调 api 公开方法。渲染层以 rAF 独立跑，读原始 Flight 对象做平滑插值（main.js:288-306）。

### 3.3 安全模型

纯前端应用：无账户、无后端、无任何网络 IO（grep WebSocket/fetch/XHR → 0 hits）。持久化仅 localStorage 语言键。调试全局（__api/__step 等 12 个）有意暴露在 window 上，任何访客可经控制台改仿真参数——按演示产品定位属接受的开放面（详见规划文档 D6）。

|---|---|
| FR-1.1 | 航班状态机 7 态：TAXIING_IN→AT_GATE→PUSHBACK→TAXIING_OUT→HOLDING→TAKEOFF→DONE，转换由航路点 tag 触发（flight-manager.js:15-23, 243-263） |

## 6. 接口与数据契约

### 6.1 标准快照 getSnapshot()（airport-api.js:368-410）

顶层：schemaVersion '1.0'、simTimeSec、wallClock、activeRunways、groundStop、metering、disruptions{weather 0-3, weatherKey, runwaysClosed, sepFactor, active}、flights[]、gates[]、runways[]（含 waiting/rolling/closed/sepFactor）、stats{arrivals, departures, onGround, gateUtil, throughput}。

每航班字段：id、callsign、airline、type、state、gate、runway、slot、position{x,y,z}、headingDeg、speedMps（×8 换算）、altitudeM、milestones{}、holdingAtGate、pobtSim、turnAtRisk、stand{contact,wide,classMatch,score}、wakeCat、eta/sta/timeToLose/seqIdx、turnaround{totalSec,t,overall,nodes[13]}。

### 6.2 事件总线（16 类，grep emit 全量核验）

| 事件 | 载荷要点 | 消费方 |
|---|---|---|
| flight_spawned / flight_arrived / atc_hold / flight_takeoff / flight_departed | getStatus() | 日志、KPI、时间线抓取 |
| no_gate | {callsign} | 溢出计数 + 警告日志 |
| tsat_release | getStatus()+heldSec（仅扣留>3 s） | 日志 |
| oooi | {code,callsign,gate,runway,sim,wall} | OOOI 流 |
| rimcas_alert | {runway,stage,sim} | 日志、回放事件轨 |
| weather_set / runway_closed / runway_opened / ground_stop / resume / metering_on / metering_off | 各自参数 | 日志、回放事件轨 |

通配 '*' 订阅机制存在但全库无订阅者；注释提及的 flight_landed 从未发射（见 KI-6）。

### 6.3 公开 API 与调试暴露

AirportAPI 24 个公开方法（spawnArrival/update/getSnapshot/getArrivalLadder/getTurnaroundWall/getStandPlan/setWeather/closeRunway/setMetering/groundStop 等，全表见 audit 清单 3.4）。window 调试暴露 12 个：__snapshot（对外数据出口）、__api、__ui、__scheduler、__analytics、__safetynet、__dcb、__guidance、__gi、__bridges、__toggleLang、__step/__pump（全管线锁步快进，测试夹具）。

### 6.4 运行日志导出

toJSON() = { meta{app, schemaVersion, startedWall, exportedWall}, events[], snapshots[], turnarounds[], oooi[] }；下载名 airport-twin-log-时间戳.json。回放接口 span()/frameAt(simT)/getIncidents() 供 RECALL 使用。

## 8. 部署与分发

### 8.1 运行形态

纯静态目录（index.html + 三层 js），任何静态服务器可跑。本地开发用 .devserver.py（强制 no-store 防 ES Module 缓存）。生产：nginx 静态服务于 opcstudio.cc（Cloudflare 前置）＋主站 config.json 注册模块入口——此两条为过程性事实，非代码可核实。

### 8.2 发布流水线（真实步骤与坑）

- 改动文件逐个 scp 至服务器 airport-twin 目录 → curl 线上 200 验证 → 同步本地 0_publicfolder 镜像 → git commit + push GitHub
- 坑一：Cloudflare 边缘缓存无版本参数的 js——验证新代码要用 cache-bust 查询串或直接查服务器文件系统，否则误判"没部署上"
- 坑二：ES Module 嵌套 import 会被浏览器缓存，本地必须用 no-store 开发服务器
- 坑三：主站 config.json 必须走 deploy-config.sh（历史上直接 scp 曾致主页宕机）

### 8.3 SEO 与站点集成

canonical、hreflang×3、OG 全套、Twitter card、WebApplication JSON-LD、关键词 title/description 均在 index.html:6-27；顶部导航栏返回主页；About 面板双语并回链 GitHub 与主站。

## 10. 质量保障与已知问题

### 10.1 质量手段（实际在用）

- __step 全管线锁步快进：控制台可复现任意时长运行，KPI 直接回读——事实上的测试夹具（main.js:352-363）
- 无头 Node 冒烟：control/optimization 层为纯 ES Module，可直接 node import 驱动验证（开发过程惯例，仓库未收录脚本）
- 每轮迭代的对抗性代码审查 + 浏览器实测（过程性惯例，非仓库内工具）
- 仓库内零自动化：无测试文件、无 CI、无 lint/formatter 配置（find → 0）

### 10.2 已知问题（全部代码核实）

| 编号 | 级别 | 问题 / 触发条件 / 根因 |
|---|---|---|
| KI-1 | 中 | 吞吐/时 KPI 混时基：唯一用墙钟（Date.now 过滤 60 分钟）的指标，__step 快进时被人为抬高；其余 KPI 均 sim 时基。根因 airport-api.js:198,219-220 |
| KI-2 | 中 | 后台标签页时间丢失：浏览器节流 setInterval 至分钟级，每次唤醒 dt 钳 1.0 s → 挂后台数小时 sim 仅前进数分钟，无任何提示。根因 main.js:216-218 |
| KI-3 | 中 | 机位全满时 spawnArrival 返回 null，该航班被整架静默丢弃（不排队不重试），仅计数+一行日志。根因 airport-api.js:127-131 |
| KI-4 | 低 | 四条环形缓冲静默丢最旧数据；回放窗口随之缩短且无提示。README"records all running data"与此不符 |
| KI-5 | 低 | README Project layout 漂移：漏列 arrival-manager、dcb-forecaster、safety-nets、guidance-lights、surface-radar 五个后加文件 |
| KI-6 | 低 | 注释漂移：airport-api.js:6 示例事件 flight_landed 从未发射（grep → 0） |
| KI-7 | 低 | 仓库残留 __overlap_test.html（手工目视页，无断言），未被 .gitignore 排除 |
| KI-8 | 低 | MAX_QUEUE=4 仅定义导出、从未生效——离港队列实际无上限。根因 runway-controller.js:14 |
| KI-9 | 中 | 除语言外一切设置刷新即丢（无持久化、无 URL 参数）——演示中途误刷新需全部重设 |
| KI-10 | 中 | 零错误上报面：无 onerror/unhandledrejection/console.error；仅有的 3 处 try/catch 均空吞（i18n.js）。任何运行时异常对用户完全静默 |
| KI-11 | 低 | DONE 航班清理用墙钟 setTimeout 3000 ms，与 sim 时钟脱钩（快进时 FIDS 残留） |
