## 1. 分析方法

**五元组**：每条 Gap = 现状（带证据）／目标（引 PR）／差距陈述／级别／关闭验收（破坏性）。
**关闭纪律**：Gap 只在破坏性验收通过后关闭，代码合入不算。
**分级**：

| 级别 | 含义 | 判据 |
|---|---|---|
| L1 品类硬伤 | 叫不叫得上这个品类名 | 品类核心承诺不成立 |
| L2 任务阻断 | 核心任务走不通 | 任务成功条件缺前提 |
| L3 健壮性 | 静默失败/失效不可见 | 失效发生时无人知晓 |
| L4 体验 | 可用但别扭 | 有绕行路径 |
| L5 工程债 | 用户不可见，拖慢演进 | 返工率/残留/不可测 |

方案细节不在本文重复——引规划文档 §9 各专项。

## 6. 依赖与风险（跨条目联动）

| 风险/联动 | 影响 | 缓解 |
|---|---|---|
| **URL 参数（G-DEMO-1）与录制导入（G-CAT-3）共同引入本产品第一个外部输入面** | 现状"数据全内生故 XSS 低危"的前提失效 | 两项必须与转义/白名单校验同批交付（规划 §7 已列）；E1 验收含恶意参数用例 |
| 自托管依赖（G-DEMO-2）改变部署面 | scp 脚本需同步 vendor 目录；与 Cloudflare 缓存坑同域 | 资源版本化（文件名带 hash）一并解决缓存误判 |
| G-ENG-1 是所有后续 Gap 的验收承载 | E0 若跳过，E1/E2 每项验收退回手工，量级全部上调 | E0 强制先行（入口条件链） |
| 时基统一（G-SIM-1）是回放/DCB 等一切时间语义的地基 | 不先修，后续时间类功能继续在错基上叠加 | 排 E0 |
| Cloudflare 边缘缓存 | 部署后误判"没生效"反复重传 | 验证一律 cache-bust（规格 §8.2 坑一固化入发布清单） |
